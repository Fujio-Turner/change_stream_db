# Failure Analysis: SOURCE → PROCESS → OUTPUT (HTTP)

> **Scope:** Every logical failure point from the moment a `_changes` feed connects to the source, through document processing, to the final HTTP request against a downstream REST endpoint.
>
> **Legend:**  
> ✅ = Working as designed  |  ⚠️ = Partially working / needs improvement  |  ❌ = Not working / missing

---

## Stage 1: SOURCE (Connecting to `_changes` Feed)

| # | Error | What **Should** Happen | What **Actually** Happens | Options / Fixes / Questions | Resolution |
|---|-------|------------------------|---------------------------|----------------------------|------------|
| 1.1 | **Source unreachable at startup** (connection refused, DNS failure, timeout) | Retry with exponential backoff forever until source comes up. Log each attempt. | ✅ `RetryableHTTP` retries up to `max_retries` (default 5) with backoff up to 60s. Feed loop (`_catch_up_normal`, `_consume_continuous_stream`, `_consume_websocket_stream`) wraps this in `while not shutdown_event` so it retries forever at the feed level. Logged as `"Catch-up request failed (attempt #N)"`. | Working. Consider: should startup specifically log `"Waiting for source to become available"` vs generic retry messages? | ✅ **Fixed.** Added a one-time `WARNING`-level log `"Source unreachable — waiting for source to become available (will retry with backoff)"` on the first failure (`failure_count == 1`) in all three feed consumers (`_catch_up_normal`, `_consume_continuous_stream`, `_consume_websocket_stream`). This gives operators an immediate, clear signal that the source is down at startup, distinct from the per-attempt `ERROR` lines that follow. The generic retry messages are preserved for ongoing diagnostics. |
| 1.2 | **Source goes down mid-stream** (TCP reset, WebSocket close, stream EOF) | Reconnect with backoff. Don't advance checkpoint. Resume from last saved `since`. | ✅ Each feed consumer catches `aiohttp.ClientError` / `asyncio.TimeoutError`, increments `failure_count`, calls `_sleep_with_backoff()`, and reconnects. Checkpoint is only advanced after successful batch processing. | Working. | ✅ **Improved.** Added a one-time `WARNING`-level log `"Source disconnected mid-stream — will reconnect with backoff"` on the first mid-stream failure (`failure_count == 1`) in both `_consume_continuous_stream` and `_consume_websocket_stream`. This gives operators an immediate, distinct signal that the source dropped the connection, separate from the per-error detail lines. Consistent with the 1.1 pattern. |
| 1.3 | **Auth failure** (401/403 from source) | Fail fast — don't retry, this won't fix itself. Log clearly. | ✅ `RetryableHTTP.request()` raises `ClientHTTPError` for permanent 4xx (401/403 classified as `"auth_failure"` by `classify_http_status()`). Pipeline crashes with clear error. | Working. **Question:** Should the pipeline enter a "paused" state instead of crashing, so a user can fix credentials and resume without losing position? | ✅ **Fixed.** Three changes: **(1)** All three feed consumers (`_catch_up_normal`, `_consume_continuous_stream`, `_consume_websocket_stream`) now log a clear auth-specific message: `"Authentication failed (HTTP 401/403) — pipeline will stop. Fix credentials and restart the job manually."` instead of a generic "non-retryable error" message. **(2)** WebSocket consumer now detects `WSServerHandshakeError` with 401/403 and raises `ClientHTTPError` instead of retrying forever. **(3)** `Pipeline` sets `_auth_failure = True` on auth crashes, exposed in `get_state()`. `PipelineManager._monitor_threads` checks this flag and **skips auto-restart** for auth failures, logging a one-time `AUTH_FAILURE` event. The job stays in error state with checkpoint preserved until credentials are fixed and the job is restarted manually. |
| 1.4 | **Source returns invalid JSON** (corrupted response, HTML error page) | Log the bad payload. Skip the batch. Retry the request. | ⚠️ `_json_loads` (orjson/json) raises `JSONDecodeError`. In continuous stream: `_parse_line()` catches it and returns `None` (skipped). In longpoll: caught by the outer `except` in `RetryableHTTP.request()` — but this counts as a permanent error, not a retry. In WebSocket: `_parse_ws_msg()` logs warning and returns empty list. | Longpoll JSON errors should be retried (source might have returned an HTML error page during a restart). Consider adding a `"source_json_errors_total"` metric. | ✅ **Fixed.** Added a `try/except (json.JSONDecodeError, ValueError)` around `_json_loads(raw_body)` in `_catch_up_normal`. On invalid JSON: logs a `WARNING` with the response length and first 200 bytes (so operators can see if it's an HTML error page), increments `stream_parse_errors_total` and `poll_errors_total` metrics, releases the response, and retries with backoff via `_sleep_with_backoff()`. Continuous stream (`_parse_line`) and WebSocket (`_parse_ws_msg`) were already handling this correctly — they skip unparseable lines and increment `stream_parse_errors_total`. |
| 1.5 | **WebSocket idle timeout** (no data for `ws_idle_timeout` seconds) | Reconnect — server may have dropped us silently. | ✅ `asyncio.wait_for(ws.receive(), timeout=ws_idle_timeout)` triggers `TimeoutError`, logs `"WebSocket idle timeout"`, increments `failure_count`, reconnects with backoff. | Working. | ✅ No changes needed. Working as designed. |
| 1.6 | **Source returns 5xx** (502/503/504 — behind a load balancer) | Retry with backoff. These are transient. | ✅ `RetryableHTTP` retries on status codes in `retry_on_status` (default: 500, 502, 503, 504, 507). Additionally, `classify_http_status()` marks all of `_TRANSIENT_5XX` as transient, so even codes not in the explicit list (e.g., 507) are retried. | Working. | ✅ No changes needed. Working as designed. |

| 2.7 | **Attachment processing fails** | Log error and either skip or DLQ the doc. | ✅ `attachment_processor.process()` exceptions are caught in `_process_one_inner`. Pipeline continues processing remaining docs. | ✅ **Fixed.** Same pattern as 2.6. Wrapped `attachment_processor.process()` in `_process_one_inner` with a `try/except Exception`. | 2.8 | **Serialization format not available** (e.g., `output_format: msgpack` but `msgpack` library not installed) | Fail fast at startup with a clear message. ## Stage 3: OUTPUT (HTTP Request to Downstream REST Endpoint)

| # | Error | What **Should** Happen | What **Actually** Happens | Options / Fixes / Questions | Resolution |
|---|-------|------------------------|---------------------------|----------------------------|------------|
| 3.1 | **Output endpoint unreachable at startup** (`test_reachable()` — DNS failure, connection refused) | Log clearly. Retry with backoff. Don't crash the pipeline before it starts. | ⚠️ `test_reachable()` catches `ClientConnectorError`, `ClientSSLError`, `TimeoutError`, and generic `ClientError`. Returns `False`. But the caller's behavior depends on pipeline config — some setups may proceed without a reachability check, others may crash. | **Question:** Should `test_reachable()` failure block pipeline startup with retry, or should the pipeline start and let per-doc retries handle it? On first failure logs a one-time `WARNING`: `"HTTP output endpoint unreachable — waiting for endpoint to become available (will retry with backoff)"`. Subsequent retries logged at `DEBUG`. The pipeline thread stays alive — no PipelineManager restart churn. Once the endpoint responds, normal feed processing begins. |
| 3.2 | **Output endpoint goes down mid-batch** (connection refused / reset during `send()`) | Hold checkpoint. Retry with backoff. Eventually backoff to 5 min. The `RetryableHTTP` layer retries 3 times with backoff up to 30s before the error reaches `send()`. | Working. | ✅ No changes needed. Working as designed. |
| 3.3 | **Output endpoint returns transient 5xx** (500, 502, 503, 504, 507) | Retry with backoff. After exhaustion, `ServerHTTPError` is raised → `send()` catches it → `OutputEndpointDown` if `halt_on_failure=true`, else returns error result for DLQ. | Working. | ✅ No changes needed. Working as designed. |
| 3.4 | **Output endpoint returns permanent 5xx** (501 Not Implemented, 505, 506, 508, 510, 511) | Don't retry — server can't handle this request type. When `halt_on_failure=true`, the pipeline halts on the first occurrence (existing behaviour). When `halt_on_failure=false`, the log gives operators an immediate signal that DLQ entries are piling up due to a config issue, not transient failures. |
| 3.5 | **Output endpoint returns permanent 4xx** (400, 401, 403, 409, 410, 413, 415, 422) | Don't retry. Batch processor writes to DLQ. | Working. | ✅ No changes needed. Working as designed. |
| 3.6 | **Auth failure on output** (401/403 from downstream endpoint) | Permanent error per-doc. Don't retry (same creds will fail again). DLQ or halt. | ✅ 401/403 are in `_PERMANENT_4XX`. ## Cross-Cutting: Checkpoint & Data Safety

| # | Error | What **Should** Happen | What **Actually** Happens | Options / Fixes / Questions | Resolution |
|---|-------|------------------------|---------------------------|----------------------------|------------|
| 4.1 | **Checkpoint save fails** (SG unreachable during checkpoint write) | Retry checkpoint save. If it fails, the worst case is re-processing some docs on restart (at-least-once). | ⚠️ `checkpoint.save()` calls `RetryableHTTP.request()` which retries 5 times. If still failing, the exception propagates. In sequential mode, this can crash the pipeline. In longpoll mode, the since value is updated in memory but not persisted — safe until restart. | **Question:** Should checkpoint save failure be non-fatal? The pipeline can continue with the in-memory `since` and try saving again on the next batch. | ✅ **Fixed.** Wrapped `checkpoint.save()` in a `try/except` in the batch processor. On failure: logs a `WARNING`: `"Checkpoint save failed — continuing with in-memory since={since}. Will retry on next batch. Worst case on restart: re-process {N} docs (at-least-once)."` The in-memory `since` value is preserved and the pipeline continues processing. A `_checkpoint_save_pending` flag is set so the next successful batch attempts to save again. Metric `checkpoint_save_failures_total` added. The pipeline no longer crashes on transient SG unavailability during checkpoint writes. Consistent with RDBMS pattern (FAILURE_OPTION_OUTPUT_RDBMS.md §4.1). |
| 4.2 | **Pipeline crash + restart = re-processing** | Docs processed since last checkpoint are re-processed. Downstream endpoint should handle duplicates (idempotent PUT). | ✅ By design. Checkpoint is per-job (`checkpoint::{job_uuid}`). Re-processing sends duplicate HTTP requests. For PUT-based endpoints this is idempotent. For POST endpoints, duplicates may create duplicate resources. | Working. **Reminder:** POST-based endpoints will get duplicates on re-processing. Use PUT (idempotent) or ensure the downstream endpoint has its own deduplication. | ✅ **Documented.** Added documentation in output config: `"Re-processing after a crash sends duplicate HTTP requests. PUT endpoints handle this idempotently. POST endpoints should either support Idempotency-Key headers or have server-side deduplication. Set write_method: PUT when possible."` Also linked from the 3.17 resolution for parallel mode. |
| 4.3 | **DLQ write fails** (CBL store unavailable) | Log the DLQ write failure. Don't lose the original doc. | ⚠️ `dlq.write()` catches internal exceptions and logs them, but the `result` is still marked as failed. In `halt_on_failure: false` mode, the doc is effectively lost (DLQ failed, checkpoint advances). In `halt_on_failure: true` mode, the batch fails and will be retried (safe). | For `halt_on_failure: false`: should DLQ write failure upgrade to a halt? | ✅ **Fixed.** Addressed by the 3.16 fix. When `halt_on_failure: false` and a DLQ write fails, the batch processor upgrades to a halt: raises `OutputEndpointDown`, checkpoint does not advance, and the doc will be re-processed on the next attempt. `CRITICAL`-level log and `dlq_write_failures_total` metric ensure visibility. | 4.4 | **At-least-once delivery guarantee** | Every doc is delivered at least once. Duplicates are possible. Documentation updated with the trade-off: `"halt_on_failure: true (default) = at-least-once guaranteed, pipeline blocks on output failure. halt_on_failure: false = at-least-once guaranteed via DLQ safety net — if both output and DLQ fail, pipeline halts to prevent data loss."` |

## HTTP-Specific: Status Code Classification

The HTTP output module classifies every response status via `classify_http_status()`. This table summarizes the behavior:

| Category | Status Codes | Behavior | Error Class |
|----------|-------------|----------|-------------|
| **Success** | 2xx | ✅ Continue | `"success"` |
| **Redirect** | 3xx | Error (unless `follow_redirects=true`) | `"redirect"` |
| **Transient 4xx** | 404, 408, 421, 423, 425, 429 | Retry with backoff | `"not_found"`, `"client_timeout"`, `"rate_limited"`, `"locked"`, etc. |
| **Permanent 4xx** | 400, 401, 402, 403, 405, 406, 407, 409, 410, 411–417, 422, 426, 428, 431, 451 | DLQ immediately (no retry) | `"auth_failure"`, `"conflict"`, `"bad_request"`, `"gone"`, `"payload_too_large"`, etc. |
| **Transient 5xx** | 500, 502, 503, 504, 507 | Retry with backoff | `"service_unavailable"`, `"bad_gateway"`, `"gateway_timeout"`, etc. |
| **Permanent 5xx** | 501, 505, 506, 508, 510, 511 | DLQ immediately (no retry) | `"server_permanent:{status}"` |
| **Unknown 4xx** | Other 4xx | Assume permanent | `"client_error:{status}"` |
| **Unknown 5xx** | Other 5xx | Assume transient (retry) | `"server_error:{status}"` |

## Performance Optimizations (Do Less / Do It Less Often / Do It Faster)

> **Philosophy:** *"There are only three optimizations: Do less. Do it less often. Do it faster. The largest gains come from #1, but we spend all our time on #3."*

### Applied Optimizations

| # | Category | What Changed | File | Details |
|---|----------|-------------|------|---------|
| OPT-1 | 🥇 Do Less | **Log "no mapping" once, not per-doc** | `rest/changes_http.py` — `_process_one_inner` | When no mapping filter matches, every doc triggered a log message. In multi-type collections where most docs don't match, this generated massive log noise. **Fix:** First unmatched doc logs a one-time `WARNING`. Subsequent docs increment `mapper_skipped_total` silently. |
| OPT-2 | 🥇 Do Less | **Skip serialization for dry-run** | `rest/output_http.py` — `send()` | In dry-run mode, the doc was serialized (potentially expensive for XML/BSON) even though the result was only logged, not sent. **Fix:** Dry-run skips `serialize_doc()` and logs the raw doc size estimate instead. |
| OPT-3 | 🥈 Do It Less Often | **Backpressure baseline is computed once** | `rest/changes_http.py` — `_maybe_backpressure()` | The baseline latency is computed from the first 50 samples and stored permanently on the `metrics` object. Subsequent calls compare against the stored baseline — no recomputation. |
| OPT-4 | 🥈 Do It Less Often | **Health check uses dedicated timeout** | `rest/output_http.py` — `_health_check()` | Health probes use `hc_timeout` (default 5s), not the main `request_timeout_seconds` (default 30s). This prevents slow health checks from blocking the event loop. |

### Remaining Optimization Opportunities (Not Yet Applied)

| # | Category | Opportunity | Impact |
|---|----------|-------------|--------|
| OPT-R1 | 🥈 Do It Less Often | **Checkpoint every N docs or T seconds** instead of every batch. High-throughput continuous streams with small batches (1–5 docs) write a checkpoint HTTP request per batch. `every_n_docs` config exists but defaults to per-batch. | Medium — reduces SG write load proportional to batch frequency. |
| OPT-R2 | 🥈 Do It Less Often | **Backoff: don't reset to zero on brief success.** When the source flaps (up 1s, down 1s), backoff resets on each brief success then ramps up again, causing oscillating reconnect storms. | Low–Medium — only affects flapping sources. |
| OPT-R3 | 🥇 Do Less | **Connection pooling for output.** `RetryableHTTP` shares the `aiohttp.ClientSession` (which has its own connection pool), but the pool size is not tuned per-output. For high-throughput parallel mode, a larger pool reduces connection setup overhead. | Medium — helps parallel mode throughput. |
| OPT-R4 | 🥈 Do It Less Often | **Batch HTTP requests.** For endpoints that support batch APIs (e.g., `POST /batch` with array body), coalesce N docs into one HTTP request. Currently each doc is one HTTP request. | High — reduces round-trips by `batch_size×`. Requires endpoint-specific config. |
