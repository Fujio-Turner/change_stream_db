## Stage 1: SOURCE (Connecting to `_changes` Feed)

| # | Error | What **Should** Happen | What **Actually** Happens | Options / Fixes / Questions |
|---|-------|------------------------|---------------------------|----------------------------|
| 1.1 | **Source unreachable at startup** (connection refused, DNS failure, timeout) | Retry with exponential backoff forever until source comes up. ## Stage 2: PROCESS (Document Fetch, Filter, Eventing, Mapping)

| # | Error | What **Should** Happen | What **Actually** Happens | Options / Fixes / Questions |
|---|-------|------------------------|---------------------------|----------------------------|
| 2.1 | **Doc fetch fails** (`include_docs=false`, GET for full doc returns error) | Retry the individual doc fetch with backoff. | 2.6 | **Eventing handler raises unexpected exception** | DLQ the doc? | 2.7 | **Attachment processing fails** | Log error and either skip or DLQ the doc. | âťŚ `attachment_processor.process()` exception is **re-raised** (`raise`). | 2.9 | **Serialization failure** (doc cannot be serialized to selected format â€” TypeError, ValueError, OverflowError) | Return error result. DLQ the doc. | 2.10 | **Doc matches no mapping filter** | Skip silently. This is expected for multi-type collections. | âś… Returns `{ok: true, skipped: true}`. Metric: `mapper_skipped_total`. | Working. |
| 2.11 | **Recursion guard suppresses doc** (doc is an echo of our own write-back) | Skip silently. Don't send to output. | âś… Returns `{ok: true, skipped: true, recursion_suppressed: true}`. Metric: `recursion_guard_suppressed_total`. | Working. |

## Stage 3: OUTPUT (HTTP Request to Downstream REST Endpoint)

| # | Error | What **Should** Happen | What **Actually** Happens | Options / Fixes / Questions |
|---|-------|------------------------|---------------------------|----------------------------|
| 3.1 | **Output endpoint unreachable at startup** (`test_reachable()` â€” DNS failure, connection refused) | Log clearly. Retry with backoff. Don't crash the pipeline before it starts. | âš ď¸Ź `test_reachable()` catches `ClientConnectorError`, `ClientSSLError`, `TimeoutError`, and generic `ClientError`. Returns `False`. But the caller's behavior depends on pipeline config â€” some setups may proceed without a reachability check, others may crash. | **Question:** Should `test_reachable()` failure block pipeline startup with retry, or should the pipeline start and let per-doc retries handle it? Currently the pipeline can start even if the output is unreachable, which leads to immediate `OutputEndpointDown` on the first doc. |
| 3.2 | **Output endpoint goes down mid-batch** (connection refused / reset during `send()`) | Hold checkpoint. Retry with backoff. Eventually backoff to 5 min. The `RetryableHTTP` layer retries 3 times with backoff up to 30s before the error reaches `send()`. | Working. |
| 3.3 | **Output endpoint returns transient 5xx** (500, 502, 503, 504, 507) | Retry with backoff. After exhaustion, `_ServerHTTPError` is raised â†’ `send()` catches it â†’ `OutputEndpointDown` if `halt_on_failure=true`, else returns error result for DLQ. | Working. |
| 3.4 | **Output endpoint returns permanent 5xx** (501 Not Implemented, 505, 506, 508, 510, 511) | Don't retry â€” server can't handle this request type. | 3.5 | **Output endpoint returns permanent 4xx** (400, 401, 403, 409, 410, 413, 415, 422) | Don't retry. | 3.6 | **Auth failure on output** (401/403 from downstream endpoint) | Permanent error per-doc. Don't retry (same creds will fail again). DLQ or halt. | âś… 401/403 are in `_PERMANENT_4XX`. Classified as `"auth_failure"`. `_ClientHTTPError` raised immediately. `send()` returns error result with `error_class: "auth_failure"`. Metrics: `outbound_auth_failure_total`. | Working. **Question:** Should auth failures halt the pipeline regardless of `halt_on_failure`? If creds are wrong, every doc will fail. Currently each doc generates a DLQ entry with `halt_on_failure=false`. |
| 3.7 | **Rate limiting** (429 Too Many Requests) | Retry with backoff. Respect `Retry-After` header if present. | âš ď¸Ź 429 is in `_TRANSIENT_4XX`. Classified as `"rate_limited"`. `RetryableHTTP.request()` retries with standard exponential backoff. | **Fix:** `Retry-After` header is **not respected**. The retry uses fixed exponential backoff instead of the server's requested delay. For 429 responses, `Retry-After` should be parsed and used as the minimum backoff delay. |
| 3.8 | **Request timeout** (`request_timeout_seconds` exceeded) | Retry. The endpoint may be slow but alive. | âš ď¸Ź `asyncio.TimeoutError` is caught by `send()`. If `halt_on_failure=true` â†’ raises `OutputEndpointDown`. But the timeout in `send()` is on the **outer** call, not inside `RetryableHTTP`. Inside `RetryableHTTP.request()`, `asyncio.TimeoutError` triggers retries (up to `max_retries`). Only after exhaustion does the error propagate to `send()`. | Working at the RetryableHTTP layer. **Question:** The `request_timeout_seconds` (default 30s) is set on `aiohttp.ClientTimeout(total=...)` per-request inside the retry loop â€” so each individual attempt has the full timeout. Is 30s appropriate for all doc sizes? Large docs (with attachments) may need longer. |
| 3.9 | **SSL/TLS handshake failure** (cert expired, mismatch, self-signed) | If `accept_self_signed_certs=false` (default): permanent error, clear message. If `halt_on_failure=true` â†’ `OutputEndpointDown`. Config option `accept_self_signed_certs` creates an SSL context with `check_hostname=False` and `CERT_NONE`. | Working. |
| 3.10 | **Invalid URL** (malformed `url_template` or `target_url`) | Permanent error per-doc. Don't retry. | âś… `aiohttp.InvalidURL` caught in `send()`. | 3.11 | **Redirect (3xx)** from output endpoint | If `follow_redirects=true`: follow automatically. When not following: `RetryableHTTP.request()` raises `RedirectHTTPError`. `send()` catches `_RedirectHTTPError` â†’ `OutputEndpointDown` if `halt_on_failure=true`, else returns error result. | Working. |
| 3.12 | **Payload too large** (413 from endpoint) | Permanent error. DLQ the doc. | âś… 413 is in `_PERMANENT_4XX`. Classified as `"payload_too_large"`. Raised as `ClientHTTPError`, caught by `send()`, routed to DLQ. | Working. **Question:** Should large docs be automatically chunked or compressed before sending? Currently no compression support (`Content-Encoding: gzip` not offered). |
| 3.13 | **Content type mismatch** (415 Unsupported Media Type) | Permanent error. This means `output_format` doesn't match what the endpoint accepts. | âś… 415 is in `_PERMANENT_4XX`. | 3.14 | **Conflict** (409 from endpoint) | Permanent error per-doc. DLQ it. | âś… 409 classified as `"conflict"` (permanent). Not retried. | 3.15 | **`halt_on_failure: true`** (default) â€” retries exhausted for transient errors | Raise `OutputEndpointDown`. Stop the batch. Don't advance checkpoint. | 3.16 | **`halt_on_failure: false`** â€” retries exhausted | Return error result. DLQ the doc. Continue with next doc in batch. | 3.17 | **Parallel mode + output down** (multiple docs in-flight) | One `OutputEndpointDown` should stop all in-flight tasks. | 3.18 | **Output backpressure** (endpoint responding slowly, latency exceeds baseline) | Slow down processing to avoid overwhelming the endpoint. | âś… `_maybe_backpressure()` monitors rolling average response time. | 3.19 | **Health check fails** (periodic heartbeat probe returns error) | Update metrics. | 3.20 | **DELETE with no body** (`send_delete_body: false`, default) | Send DELETE without request body. Some APIs require this. | âś… When `method == "DELETE" and not self._send_delete_body`, the `data` key is omitted from request kwargs. `Content-Type` header is also not set. | Working. |

## Cross-Cutting: Checkpoint & Data Safety

| # | Error | What **Should** Happen | What **Actually** Happens | Options / Fixes / Questions |
|---|-------|------------------------|---------------------------|----------------------------|
| 4.1 | **Checkpoint save fails** (SG unreachable during checkpoint write) | Retry checkpoint save. If it fails, the worst case is re-processing some docs on restart (at-least-once). | âš ď¸Ź `checkpoint.save()` calls `RetryableHTTP.request()` which retries 5 times. If still failing, the exception propagates. In sequential mode, this can crash the pipeline. In longpoll mode, the since value is updated in memory but not persisted â€” safe until restart. | **Question:** Should checkpoint save failure be non-fatal? The pipeline can continue with the in-memory `since` and try saving again on the next batch. |
| 4.2 | **Pipeline crash + restart = re-processing** | Docs processed since last checkpoint are re-processed. Downstream endpoint should handle duplicates (idempotent PUT). | âś… By design. Checkpoint is per-job (`checkpoint::{job_uuid}`). Re-processing sends duplicate HTTP requests. For PUT-based endpoints this is idempotent. For POST endpoints, duplicates may create duplicate resources. | Working. **Reminder:** POST-based endpoints will get duplicates on re-processing. Use PUT (idempotent) or ensure the downstream endpoint has its own deduplication. |
| 4.3 | **DLQ write fails** (CBL store unavailable) | Log the DLQ write failure. Don't lose the original doc. | âš ď¸Ź `dlq.write()` catches internal exceptions and logs them, but the `result` is still marked as failed. In `halt_on_failure: false` mode, the doc is effectively lost (DLQ failed, checkpoint advances). In `halt_on_failure: true` mode, the batch fails and will be retried (safe). | For `halt_on_failure: false`: should DLQ write failure upgrade to a halt? |
| 4.4 | **At-least-once delivery guarantee** | Every doc is delivered at least once. Duplicates are possible. Never lose a doc silently. | âš ď¸Ź Guarantee holds when `halt_on_failure: true` â€” checkpoint doesn't advance past failed docs. When `halt_on_failure: false` with DLQ failure (4.3), docs can be silently lost. | Document the `halt_on_failure` trade-off clearly: `true` = at-least-once guaranteed but pipeline blocks on output failure; `false` = best-effort with DLQ, docs may be lost if both output and DLQ fail. |

## HTTP-Specific: Status Code Classification

The HTTP output module classifies every response status via `classify_http_status()`. This table summarizes the behavior:

| Category | Status Codes | Behavior | Error Class |
|----------|-------------|----------|-------------|
| **Success** | 2xx | âś… Continue | `"success"` |
| **Redirect** | 3xx | Error (unless `follow_redirects=true`) | `"redirect"` |
| **Transient 4xx** | 404, 408, 421, 423, 425, 429 | Retry with backoff | `"not_found"`, `"client_timeout"`, `"rate_limited"`, `"locked"`, etc. |
| **Permanent 4xx** | 400, 401, 402, 403, 405, 406, 407, 409, 410, 411â€“417, 422, 426, 428, 431, 451 | DLQ immediately (no retry) | `"auth_failure"`, `"conflict"`, `"bad_request"`, `"gone"`, `"payload_too_large"`, etc. |
| **Transient 5xx** | 500, 502, 503, 504, 507 | Retry with backoff | `"service_unavailable"`, `"bad_gateway"`, `"gateway_timeout"`, etc. |
| **Permanent 5xx** | 501, 505, 506, 508, 510, 511 | DLQ immediately (no retry) | `"server_permanent:{status}"` |
| **Unknown 4xx** | Other 4xx | Assume permanent | `"client_error:{status}"` |
| **Unknown 5xx** | Other 5xx | Assume transient (retry) | `"server_error:{status}"` |
