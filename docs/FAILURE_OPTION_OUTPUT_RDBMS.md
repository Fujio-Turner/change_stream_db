| # | Error | What **Should** Happen | What **Actually** Happens | Options / Fixes / Questions | Resolution |
|---|-------|------------------------|---------------------------|----------------------------|------------|
| 3.1 | **RDBMS unreachable at startup** (`connect()` fails ‚ÄĒ can't create pool) | Retry with backoff. Don't crash the pipeline before it starts. | ‚ĚĆ `await output.connect()` in `poll_changes()` raises immediately. Exception propagates to `Pipeline.run()` ‚Üí pipeline enters error state. PipelineManager restarts with backoff (now up to 5 min after our fix). But: there's a **log storm** because the pipeline re-runs the entire `poll_changes` setup on each restart attempt. | **Option A (recommended):** Add a connect-with-retry loop inside `poll_changes()` before entering the feed loop: retry `output.connect()` with backoff up to 300s, so the pipeline thread stays alive and doesn't churn through PipelineManager restart cycles. **Option B:** Accept the PipelineManager restart approach but suppress repeated startup log noise. | ‚úÖ **Fixed (Option A).** Added a connect-with-retry loop in `poll_changes()` before entering the feed loop. `output.connect()` is retried with exponential backoff (1s ‚Üí 2s ‚Üí 4s ‚Üí ‚Ä¶ ‚Üí 300s cap) inside a `while not shutdown_event.is_set()` loop. On first failure logs a one-time `WARNING`: `"RDBMS unreachable ‚ÄĒ waiting for database to become available (will retry with backoff)"`. Subsequent retries logged at `DEBUG`. The pipeline thread stays alive ‚ÄĒ no PipelineManager restart churn, no log storm. Once the connection succeeds, normal feed processing begins. |
| 3.2 | **RDBMS goes down mid-batch** (pool connections die ‚ÄĒ `ConnectionError`) | Hold checkpoint. Retry with backoff. Eventually backoff to 5 min. Retry forever. When RDBMS comes back up, automatically reconnect and resume processing. | ‚ö†ÔłŹ Per-doc retry loop in `db_base.py` retries `max_retries` (default 3) times with backoff up to `backoff_max_seconds` (default 10s). After exhaustion, if `halt_on_failure=true` ‚Üí raises `OutputEndpointDown`. Feed loop now backs off up to 300s (after our fix). **Problem:** Every in-flight doc independently calls `_reconnect_pool()`, causing a stampede of `TooManyConnectionsError` (visible in your logs). **Critical Problem:** The pipeline **never recovered** even after PostgreSQL came back up. | **Fix:** The `_reconnect_pool` generation check (`_pool_generation`) should prevent stampede, but the error in `_reconnect_pool` itself (TooManyConnections on pool creation) is not suppressed for concurrent callers. **Option:** Add a "pool is reconnecting" flag so only one task reconnects while others wait on the lock. Currently `_pool_lock` is per-forwarder but each doc's retry loop calls `_reconnect_pool` independently. This is already partially handled by generation check ‚ÄĒ investigate why it's still storming. | ‚úÖ **Fixed (v2.2.3).** Four bugs across three files prevented the pipeline from recovering after RDBMS came back up. **Bug 1 ‚ÄĒ Cooldown guard blocked recovery** (`db/db_postgres.py`): When `_connect_pool()` failed, `_close_pool()` had already set `self._pool = None`. On the next retry the 1-second cooldown silently returned without attempting to reconnect, leaving the pool permanently `None`. **Fix:** The cooldown check now skips when `self._pool is None` ‚ÄĒ a dead pool must always be restored regardless of cooldown. **Bug 2 ‚ÄĒ Stale generation in `send()` retry loop** (`db/db_base.py`): When `_connect_pool()` failed, `_pool_generation` was not bumped, so all subsequent `send()` calls captured the same stale `pool_gen` and got cooldown-skipped. **Fix:** `_pool_generation` is now incremented in the `except` branch so callers don't stack up against a dead pool. Additionally, `send()` now updates `pool_gen = self._pool_generation` after a **failed** reconnect (not just after success), so remaining retry attempts within the same `send()` call don't hit the stale-generation fast path and skip reconnection entirely. **Bug 3 ‚ÄĒ Parallel-mode stampede returned success on failure** (`db/db_postgres.py`): In parallel mode, when Task A's reconnect failed (pool still `None`, generation bumped), Tasks B/C/D waiting in the stampede guard saw the changed generation and returned **successfully** ‚ÄĒ tricking their callers into thinking the pool was restored. All their remaining retries silently operated on a `None` pool. **Fix:** All three generation checks in `_reconnect_pool()` (fast-path, stampede-wait, and post-lock) now require `self._pool is not None` before returning early. If the generation changed but pool is still `None`, the other task's reconnect failed ‚ÄĒ callers fall through and attempt reconnection themselves. **Bug 4 ‚ÄĒ WebSocket mode had no outer retry loop** (`main.py`): Unlike continuous mode which wraps catch-up + stream in `while not stop_event.is_set()`, WebSocket mode ran catch-up once, then the WebSocket stream once, then `return`. When the WebSocket consumer returned after a prolonged RDBMS outage, `poll_changes()` exited, `Pipeline.run()` logged "stopped cleanly" (not an error), and `PipelineManager._monitor_threads` saw `status != "error"` ‚ÄĒ so it **never restarted** the pipeline. Short outages (< 30s) recovered because the WebSocket consumer's internal retry loop reconnected the pool before the consumer exited. Longer outages exhausted the internal retries, causing the consumer to return normally. **Fix:** WebSocket mode now uses the same `while not stop_event.is_set()` outer loop as continuous mode, cycling between catch-up and WebSocket streaming indefinitely until shutdown. **Bug 5 ‚ÄĒ No proactive pool recovery** (`db/db_postgres.py`): `asyncpg` does not auto-heal a dead pool ‚ÄĒ reconnection was only attempted inside `send()` during query execution. When all `send()` retries exhausted and the feed-level backoff kicked in (up to 300s), no code was trying to restore the pool during that wait. When the backoff expired and the next batch ran, `send()` tried to reconnect but PG might still be briefly unreachable, exhausting retries again before PG fully stabilized. This created a race where the pipeline could miss the recovery window. **Fix:** Added a background health-check task (`_health_check_loop`) that starts automatically when `_reconnect_pool()` fails. The task runs independently of the feed loop, periodically attempting to recreate the pool with its own exponential backoff (2s ‚Üí 4s ‚Üí 8s ‚Üí ‚Ä¶ ‚Üí 300s). When PG comes back, the health check restores the pool proactively ‚ÄĒ so by the time the feed-level backoff expires and `send()` runs again, the pool is already alive. The task self-terminates once the pool is restored (either by itself or by a concurrent `send()` retry). Cancelled on `close()`. This matches the "Resilient Pool" pattern used by production `asyncpg` deployments. |
