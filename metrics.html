Prometheus Metric Types

Counter counter

A value that only goes up (or resets to zero on restart). Use rate() or increase() in PromQL to get per-second or per-interval rates. Never graph the raw value — it's cumulative.

Gauge gauge

A value that can go up or down. Represents a snapshot: current uptime, last batch size, endpoint health. Graph the raw value directly.

Summary summary

Pre-calculated quantiles (p50, p90, p99) plus total count and sum. Ideal for response time distributions. Use _sum / _count for the average.

Info info

A gauge that's always 1, carrying metadata in labels. Used for non-numeric values like the current checkpoint sequence string.

Labels

Every metric includes these labels for multi-instance dashboards:

LabelExampleDescription
srcsync_gatewayGateway type: sync_gateway, app_services, or edge_server
databasedbThe database name from gateway.database in config

Process Metrics

changes_worker_uptime_seconds gauge

Time in seconds since the worker process started. Resets to zero on restart.

What it tells you
How long the worker has been running continuously. A sudden drop to zero means the process restarted.
Chart suggestion
Single-stat panel showing uptime in hours/days. Alert if uptime drops below a threshold (unexpected restarts).
PromQL
changes_worker_uptime_seconds
Alert example
# Worker restarted in the last 5 minutes
changes_worker_uptime_seconds < 300

Poll Loop Metrics

changes_worker_poll_cycles_total counter

Total number of _changes poll cycles completed. Incremented after every successful response from the gateway, whether the batch contained changes or not.

What it tells you
How actively the worker is polling. If this counter stops increasing, the worker is stuck or the gateway is unreachable.
Chart suggestion
Line chart with rate() — shows polls per second. Should match roughly 1 / poll_interval_seconds when idle.
PromQL
rate(changes_worker_poll_cycles_total[5m])
changes_worker_poll_errors_total counter

Total number of _changes poll errors. Includes non-retryable errors (4xx) and exhausted retries (5xx / connection failures).

What it tells you
Gateway connectivity problems. Any non-zero rate means the worker is failing to read the feed.
Chart suggestion
Line chart with rate(). Overlay with poll_cycles_total to see error ratio. Alert if > 0 for sustained periods.
PromQL
# Error rate
rate(changes_worker_poll_errors_total[5m])

# Error ratio (errors / total polls)
rate(changes_worker_poll_errors_total[5m])
  / rate(changes_worker_poll_cycles_total[5m])
changes_worker_last_poll_timestamp_seconds gauge

Unix timestamp (epoch seconds) of the last successful _changes poll response.

What it tells you
How recently the worker successfully fetched changes. If this timestamp is stale, the worker may be stuck or the gateway is down.
Chart suggestion
Single-stat showing "time since last poll" using time() - value. Alert if the gap exceeds 2× your poll interval.
PromQL
# Seconds since last successful poll
time() - changes_worker_last_poll_timestamp_seconds
changes_worker_last_batch_size gauge

Number of changes in the most recent batch received from the _changes feed.

What it tells you
Current load. A batch size of 0 means the worker is caught up (no new changes). Large batches indicate heavy write activity or a catch-up scenario.
Chart suggestion
Line chart or bar chart. If using throttle_feed, full batches (= throttle limit) mean there's more data waiting.
PromQL
changes_worker_last_batch_size

Changes Metrics

changes_worker_changes_received_total counter

Total number of changes received from the _changes feed. This is the raw count before any filtering.

What it tells you
Total throughput from the gateway. This is your ingest rate — how many document mutations the gateway is reporting.
Chart suggestion
Line chart with rate() — shows changes per second arriving from the feed. Compare against changes_processed_total to see how many survive filtering.
PromQL
# Changes arriving per second
rate(changes_worker_changes_received_total[5m])

# Total changes in the last hour
increase(changes_worker_changes_received_total[1h])
changes_worker_changes_processed_total counter

Total number of changes that passed filtering and were forwarded to the output endpoint.

What it tells you
Your effective processing throughput — the work the output endpoint actually receives. The difference between received and processed equals filtered-out changes.
Chart suggestion
Line chart with rate(). Stack with changes_filtered_total to show the full picture.
PromQL
rate(changes_worker_changes_processed_total[5m])
changes_worker_changes_filtered_total counter

Total number of changes filtered out (sum of deletes + removes skipped due to ignore_delete / ignore_remove settings).

What it tells you
How many changes are being discarded by your filter rules. A high ratio may indicate heavy delete activity in the source database.
PromQL
# Filter ratio: what percentage of changes are dropped
rate(changes_worker_changes_filtered_total[5m])
  / rate(changes_worker_changes_received_total[5m]) * 100
changes_worker_changes_deleted_total counter

Total deleted changes filtered out (subset of changes_filtered_total). Only incremented when ignore_delete=true.

What it tells you
Delete activity in the source database that was intentionally skipped.
PromQL
rate(changes_worker_changes_deleted_total[5m])
changes_worker_changes_removed_total counter

Total removed-from-channel changes filtered out (subset of changes_filtered_total). Only incremented when ignore_remove=true.

What it tells you
How often documents are being revoked from channels. High numbers may indicate channel assignment churn.
PromQL
rate(changes_worker_changes_removed_total[5m])
changes_worker_feed_deletes_seen_total counter

Total changes with deleted=true seen in the _changes feed. Counted always, regardless of whether ignore_delete is enabled. This tells you the true delete activity in the source database.

What it tells you
How many document deletions are happening in the source, independent of your filter settings. Compare against changes_deleted_total to confirm your filter is working — if ignore_delete=false, changes_deleted_total stays at 0 while this metric still climbs.
Chart suggestion
Line chart with rate(). Overlay with changes_received_total to see what percentage of feed activity is deletions.
PromQL
# Delete ratio in the feed
rate(changes_worker_feed_deletes_seen_total[5m])
  / rate(changes_worker_changes_received_total[5m]) * 100
changes_worker_feed_removes_seen_total counter

Total changes with removed=true seen in the _changes feed. Counted always, regardless of whether ignore_remove is enabled. These are documents revoked from channels the worker is listening to.

What it tells you
Channel access revocation rate. High values may indicate frequent channel reassignment or access control changes in your Sync Function.
PromQL
rate(changes_worker_feed_removes_seen_total[5m])

Bytes / Data Volume Metrics

changes_worker_bytes_received_total counter

Total bytes received from the gateway. Includes raw response bodies from _changes feed responses, _bulk_get responses (SG/App Services), and individual document GET responses (Edge Server). This is the actual wire payload — the JSON data the worker ingests.

What it tells you
Ingest data volume. Combine with uptime or time windows to get bytes/second throughput. Large values indicate big documents or high change volume.
Chart suggestion
Line chart with rate() in bytes/sec. Apply Grafana unit formatting (bytes/sec → KB/s, MB/s) for readability.
PromQL
# Ingest throughput in bytes/sec
rate(changes_worker_bytes_received_total[5m])

# Total data ingested in the last hour (in MB)
increase(changes_worker_bytes_received_total[1h]) / 1024 / 1024

# Average document size (bytes per change received)
rate(changes_worker_bytes_received_total[5m])
  / rate(changes_worker_changes_received_total[5m])
changes_worker_bytes_output_total counter

Total bytes sent to the output endpoint. This is the serialized body size — JSON, XML, msgpack, etc. — for every document forwarded downstream.

What it tells you
Egress data volume. Compare against bytes_received_total to understand compression/expansion from serialization format changes (e.g., JSON→msgpack shrinks, JSON→XML grows).
Chart suggestion
Line chart with rate() in bytes/sec. Dual-axis with bytes_received_total to see in vs out.
PromQL
# Output throughput in bytes/sec
rate(changes_worker_bytes_output_total[5m])

# Average output payload size per doc
rate(changes_worker_bytes_output_total[5m])
  / rate(changes_worker_output_requests_total[5m])

# Input vs output ratio (should be ~1.0 for JSON→JSON)
rate(changes_worker_bytes_output_total[5m])
  / rate(changes_worker_bytes_received_total[5m])
📥

Doc Fetching Metrics

changes_worker_docs_fetched_total counter

Total documents fetched via _bulk_get (SG/App Services) or individual GET requests (Edge Server). Only incremented when include_docs=false, since the worker must fetch doc bodies separately.

What it tells you
Secondary fetch volume. If this is high, consider switching to include_docs=true to reduce round trips. For Edge Server, each doc is a separate HTTP request.
Chart suggestion
Line chart with rate(). Compare against changes_processed_total — they should roughly match when include_docs=false.
PromQL
rate(changes_worker_docs_fetched_total[5m])

Output Metrics

changes_worker_output_requests_total counter

Total output requests sent to the downstream endpoint. Includes both successful and failed requests.

What it tells you
Total delivery attempts. This counts requests sent to the configured HTTP/RDBMS/Cloud endpoint.
Chart suggestion
Line chart with rate(). This is your delivery throughput — docs per second reaching the downstream system.
PromQL
rate(changes_worker_output_requests_total[5m])
changes_worker_output_errors_total counter

Total output request errors. Includes 4xx client errors, 3xx redirects, 5xx after retries exhausted, and connection failures.

What it tells you
Downstream delivery failures. If halt_on_failure=true, a single error stops the pipeline. If false, errors mean skipped docs.
Chart suggestion
Line chart with rate(). Alert on any sustained rate > 0.
PromQL
# Output error rate
rate(changes_worker_output_errors_total[5m])

# Output success ratio
1 - (
  rate(changes_worker_output_errors_total[5m])
  / rate(changes_worker_output_requests_total[5m])
)
changes_worker_output_requests_by_method_total counter

Output requests broken down by HTTP method. Carries a method label with values "PUT" or "DELETE". PUT corresponds to document creates/updates; DELETE corresponds to document deletions.

What it tells you
The mix of create/update vs delete operations hitting your downstream endpoint. A sudden spike in DELETEs might indicate a mass purge or TTL expiration in the source.
Chart suggestion
Stacked area chart with both methods. Or a pie chart showing the PUT/DELETE ratio over a time window.
PromQL
# PUT rate
rate(changes_worker_output_requests_by_method_total{method="PUT"}[5m])

# DELETE rate
rate(changes_worker_output_requests_by_method_total{method="DELETE"}[5m])

# Delete-to-PUT ratio
rate(changes_worker_output_requests_by_method_total{method="DELETE"}[5m])
  / rate(changes_worker_output_requests_by_method_total{method="PUT"}[5m])
changes_worker_output_errors_by_method_total counter

Output errors broken down by HTTP method ("PUT" or "DELETE"). Helps pinpoint whether your downstream endpoint is failing on writes vs deletes specifically.

What it tells you
If DELETE errors are high but PUT errors are zero, your endpoint may not support DELETE. If PUT errors are high, the endpoint may be rejecting payloads (schema mismatch, size limits, etc.).
PromQL
# PUT error rate
rate(changes_worker_output_errors_by_method_total{method="PUT"}[5m])

# DELETE error rate
rate(changes_worker_output_errors_by_method_total{method="DELETE"}[5m])
changes_worker_output_endpoint_up gauge

Whether the output HTTP endpoint is reachable. 1 = up, 0 = down (set when halt_on_failure triggers).

What it tells you
Instant health signal. When this drops to 0, the worker has stopped forwarding and is holding the checkpoint.
Chart suggestion
Single-stat or status panel (green/red). Alert immediately when 0.
PromQL
changes_worker_output_endpoint_up == 0

Response Time Summary

changes_worker_output_response_time_seconds summary

Output HTTP response time distribution in seconds. Exposed as pre-calculated quantiles (p50, p90, p99) plus cumulative sum and count. Only populated when output.mode=http.

Sub-metrics
MetricDescription
{quantile="0.5"}Median response time (p50) — half of requests are faster than this
{quantile="0.9"}90th percentile (p90) — 90% of requests are faster than this
{quantile="0.99"}99th percentile (p99) — tail latency, worst 1% of requests
_sumTotal time spent on all output requests (seconds)
_countTotal number of output requests timed
What it tells you
How fast your downstream endpoint is responding. The p50 shows typical latency; p99 shows worst-case. If p99 is much higher than p50, you have tail latency issues.
Chart suggestion
Multi-line chart with all three quantiles on the same panel. Add the average (_sum / _count) as a fourth line. Y-axis in seconds (or milliseconds).
PromQL
# Average output response time
changes_worker_output_response_time_seconds_sum
  / changes_worker_output_response_time_seconds_count

# p99 directly
changes_worker_output_response_time_seconds{quantile="0.99"}

Checkpoint Metrics

changes_worker_checkpoint_saves_total counter

Total checkpoint save operations (successful PUTs to _local/checkpoint-{uuid} on the gateway).

What it tells you
How frequently the checkpoint advances. Should increase at roughly the same rate as poll_cycles_total.
PromQL
rate(changes_worker_checkpoint_saves_total[5m])
changes_worker_checkpoint_save_errors_total counter

Total checkpoint save errors. When the gateway is unreachable, the checkpoint falls back to the local checkpoint.json file.

What it tells you
Gateway write failures. If this increases, the checkpoint is being saved locally — the worker will still function but the checkpoint won't survive container restarts without volume mounts.
Chart suggestion
Line chart with rate(). Alert on any sustained rate > 0.
PromQL
rate(changes_worker_checkpoint_save_errors_total[5m]) > 0
changes_worker_checkpoint_seq info / gauge

Current checkpoint sequence value. Exposed as a gauge with value 1 and the sequence as a seq label, since sequences can be non-numeric strings (e.g., "1234:56").

What it tells you
Where the worker is in the feed. If this stops advancing, the worker is either caught up (no new changes) or stuck (output failure holding the checkpoint).
Chart suggestion
Table panel or log-style panel showing the seq label value over time.
PromQL
# See the current sequence value in the seq label
changes_worker_checkpoint_seq

Retry Metrics

changes_worker_retries_total counter

Total HTTP retry attempts across all requests (gateway _changes, _bulk_get, checkpoint, and output endpoint). Each individual retry attempt increments this counter by 1.

What it tells you
Network reliability issues. A high retry rate indicates flaky connections, overloaded servers, or transient errors. Compare against total requests to gauge severity.
Chart suggestion
Line chart with rate(). Spike correlation with poll_errors_total and output_errors_total helps identify the source.
PromQL
rate(changes_worker_retries_total[5m])

DB Transaction Metrics

changes_worker_db_retries_total counter

DB transaction retry attempts (deadlocks, serialization failures, connection drops).

changes_worker_db_retry_exhausted_total counter

Times all DB retries were exhausted — the transaction permanently failed.

changes_worker_db_transient_errors_total counter

Transient DB errors: connection drops, deadlocks, serialization failures. These are retryable.

changes_worker_db_permanent_errors_total counter

Permanent DB errors: constraint violations, type mismatches. These go to DLQ — your mappings need fixing.

changes_worker_db_pool_reconnects_total counter

DB connection pool reconnections. Spikes indicate the database is restarting or the network is flaky.

Recursion Guard Metrics

Write-back echo suppression — when the pipeline PUTs a document back to the source, the resulting _changes echo is detected and skipped to prevent infinite recursion loops.

changes_worker_recursion_guard_suppressed_total counter

PromQL — Suppression as % of Received
rate(changes_worker_recursion_guard_suppressed_total[5m])
  / rate(changes_worker_changes_received_total[5m]) * 100

Per-Engine / Per-Job DB Metrics

When using RDBMS output (Postgres, MySQL, Oracle, MSSQL), these labeled counters appear alongside the global totals. Labels: engine, job_id.

changes_worker_db_{counter_name} counter

Per-engine breakdowns of every counter the DB forwarder increments (e.g. output_requests_total, output_errors_total, db_retries_total).

Example
changes_worker_db_output_requests_total{engine="postgres",job_id="orders_sync"} 300
changes_worker_db_output_requests_total{engine="oracle",job_id="analytics"} 200
changes_worker_db_output_response_time_seconds summary

Per-engine output response time (p50, p90, p99). Labels: engine, job_id.

Derived Metrics & PRO TIPS 🧠

These are not exposed directly but can be computed in Prometheus/Grafana by combining raw metrics. These are the most valuable panels for operational dashboards.

📈 Throughput — Changes Per Second

How fast the worker is ingesting changes from the gateway.

rate(changes_worker_changes_received_total[5m])

📈 Effective Processing Rate — Docs Forwarded Per Second

How fast documents are being delivered to the downstream system, after filtering.

rate(changes_worker_changes_processed_total[5m])

⏱ Average Time Per Document (end-to-end)

Divide total output time by total docs processed to get average end-to-end processing time per document. This includes output HTTP latency.

changes_worker_output_response_time_seconds_sum
  / changes_worker_output_response_time_seconds_count

⏱ Average Time Per Poll Cycle

How long each poll cycle takes on average (from poll to next poll). Useful for understanding overall loop cadence.

changes_worker_uptime_seconds
  / changes_worker_poll_cycles_total

📊 Filter Drop Rate (%)

What percentage of incoming changes are being filtered out (deletes + removes). High values mean most feed activity is irrelevant to your pipeline.

rate(changes_worker_changes_filtered_total[5m])
  / rate(changes_worker_changes_received_total[5m]) * 100

❌ Output Error Rate (%)

Percentage of output requests that failed. Should be 0% in a healthy system.

rate(changes_worker_output_errors_total[5m])
  / rate(changes_worker_output_requests_total[5m]) * 100

📉 Poll Error Rate (%)

Percentage of poll cycles that hit errors. Indicates gateway reliability.

rate(changes_worker_poll_errors_total[5m])
  / rate(changes_worker_poll_cycles_total[5m]) * 100

⏱ Staleness — Seconds Since Last Successful Poll

How stale the data is. If this exceeds 2–3× your poll_interval_seconds, something is wrong.

time() - changes_worker_last_poll_timestamp_seconds

📊 Average Batch Size

Average number of changes per batch over time. Useful for capacity planning and understanding load patterns.

rate(changes_worker_changes_received_total[5m])
  / rate(changes_worker_poll_cycles_total[5m])

📦 Ingest Throughput — Bytes Per Second (In)

How fast data is flowing in from the gateway. Apply Grafana's bytes/sec unit for automatic KB/s, MB/s formatting.

rate(changes_worker_bytes_received_total[5m])

📦 Output Throughput — Bytes Per Second (Out)

How fast data is flowing out to your downstream endpoint.

rate(changes_worker_bytes_output_total[5m])

📦 Average Document Size (Ingest)

Average size (in bytes) of each change received from the gateway. Useful for capacity planning — tells you if docs are 500 bytes or 50KB.

rate(changes_worker_bytes_received_total[5m])
  / rate(changes_worker_changes_received_total[5m])

📦 Average Payload Size (Output)

Average bytes per output request. If you changed the output_format from JSON to msgpack, this will show the size reduction.

rate(changes_worker_bytes_output_total[5m])
  / rate(changes_worker_output_requests_total[5m])

📊 Feed Delete Rate (%)

What percentage of all changes in the feed are document deletions. Always counted, regardless of ignore_delete setting.

rate(changes_worker_feed_deletes_seen_total[5m])
  / rate(changes_worker_changes_received_total[5m]) * 100

📊 PUT vs DELETE Split

Percentage of output requests that are DELETE operations. Tells you the mutation-type distribution hitting your downstream.

rate(changes_worker_output_requests_by_method_total{method="DELETE"}[5m])
  / rate(changes_worker_output_requests_total[5m]) * 100

📊 Docs Fetched Per Processed Change

Only relevant when include_docs=false. Should be ≈1.0 — if higher, docs are being fetched redundantly.

rate(changes_worker_docs_fetched_total[5m])
  / rate(changes_worker_changes_processed_total[5m])

🧠 PRO TIP — Retry Success Rate

What % of retries eventually succeed. A value of 1.0 (100%) means every transient error recovered. If retry_exhausted_total is climbing, your max_retries is too low or the target is truly down.

1 - (
  rate(changes_worker_retry_exhausted_total[5m])
  / rate(changes_worker_retries_total[5m])
)

🧠 PRO TIP — DLQ Growth Rate

How fast the dead-letter queue is growing. Any sustained rate > 0 means docs are failing. Combine with dlq_pending_count to see the current backlog.

rate(changes_worker_dead_letter_total[5m])

🧠 PRO TIP — Attachment Download Success Rate (%)

How reliable attachment downloads are. If below 100%, check attachments_missing_total (404s) vs attachments_download_errors_total (network errors) to pinpoint the cause.

rate(changes_worker_attachments_downloaded_total[5m])
  / (rate(changes_worker_attachments_downloaded_total[5m])
     + rate(changes_worker_attachments_download_errors_total[5m])) * 100

🧠 PRO TIP — Attachment Pipeline Efficiency

What fraction of detected attachments make it through download → upload. Values < 1.0 mean attachments are being lost to filters, errors, or stale revisions. Check attachments_skipped_total, attachments_stale_total, and attachments_orphaned_uploads_total.

rate(changes_worker_attachments_uploaded_total[5m])
  / rate(changes_worker_attachments_detected_total[5m])

🧠 PRO TIP — DB Transient vs Permanent Error Ratio

If most DB errors are transient (deadlocks, connection drops), the system self-heals via retries. If most are permanent (constraint violations, type mismatches), your schema mappings need fixing.

rate(changes_worker_db_transient_errors_total[5m])
  / (rate(changes_worker_db_transient_errors_total[5m])
     + rate(changes_worker_db_permanent_errors_total[5m]))

🧠 PRO TIP — Backpressure Impact (%)

What % of the worker's lifetime has been spent throttled. If > 10%, your output endpoint is the bottleneck.

changes_worker_backpressure_delay_seconds_total
  / changes_worker_uptime_seconds * 100

🧠 PRO TIP — Memory Leak Detection

If RSS memory is growing linearly over hours, you likely have a memory leak. Combine with python_gc_gen2_count — if gen2 objects grow too, objects survive GC.

deriv(changes_worker_process_memory_rss_bytes[1h])

🧠 PRO TIP — Inbound vs Outbound Auth Failure Rate

Pinpoints which side has auth issues. If inbound failures climb → gateway credentials expiring. If outbound failures climb → output endpoint credentials need refresh.

# Inbound (gateway) auth failure rate
rate(changes_worker_inbound_auth_failure_total[5m])
  / rate(changes_worker_inbound_auth_total[5m]) * 100

# Outbound (output) auth failure rate
rate(changes_worker_outbound_auth_failure_total[5m])
  / rate(changes_worker_outbound_auth_total[5m]) * 100

Alerting Examples

Copy-paste these Prometheus alerting rules to get started:

groups:
  - name: changes_worker
    rules:

      # Worker is down or restarted
      - alert: ChangesWorkerDown
        expr: changes_worker_uptime_seconds < 60
        for: 2m
        labels:
          severity: critical
        annotations:
          summary: "Changes worker restarted or down"

      # No polls in 2× poll_interval
      - alert: ChangesWorkerStale
        expr: time() - changes_worker_last_poll_timestamp_seconds > 120
        for: 1m
        labels:
          severity: warning
        annotations:
          summary: "No successful poll in over 2 minutes"

      # Output endpoint is down
      - alert: OutputEndpointDown
        expr: changes_worker_output_endpoint_up == 0
        for: 0m
        labels:
          severity: critical
        annotations:
          summary: "Output endpoint down, checkpoint frozen"

      # Output error rate > 1%
      - alert: OutputErrorRate
        expr: >
          rate(changes_worker_output_errors_total[5m])
          / rate(changes_worker_output_requests_total[5m]) > 0.01
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "Output error rate exceeds 1%"

      # p99 latency > 1 second
      - alert: OutputLatencyHigh
        expr: >
          changes_worker_output_response_time_seconds{quantile="0.99"} > 1.0
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "Output p99 latency exceeds 1 second"

      # Checkpoint falling back to local file
      - alert: CheckpointFallback
        expr: rate(changes_worker_checkpoint_save_errors_total[5m]) > 0
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "Checkpoint saving to local file (gateway unreachable)"
